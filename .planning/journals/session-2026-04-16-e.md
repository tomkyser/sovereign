## What Was Tried

### 1. String Ordering Fix (Option A from journal)
- **Implementation**: Sorted module strings by original offset before writing to preserve bytecode→content layout
- **Result**: FAILED — same crash
- **Analysis**: Debug logging confirmed bytecode IS placed before content in the rebuilt data, but crash persists

### 2. Bytecode Clearing + Encoding=0
- **Implementation**: Set `bytecodeBytes = Buffer.alloc(0)` and `encoding = 0` for patched module

- **Why**: The source text in the binary is bytecode-compiled (`// @bun @bytecode @bun-cjs` prefix). Even without the bytecode blob, the source itself is not readable JS — function bodies are compiled stubs.

- **Result**: FAILED — same crash for the same reason

## Key Discovery: tweakcc PR #655

User pointed to https://github.com/Piebald-AI/tweakcc/pull/655 which addresses the SAME issue.

**Their approach**: When the extracted source has the `@bun @bytecode` prefix (meaning it's bytecode-compiled and function bodies are stubs), they **fetch clean readable source from npm** via `npm pack @anthropic-ai/claude-code@X.Y.Z`, extract the readable `cli.js`, patch THAT, then repack with bytecode cleared.

**Why this works**: The npm-published source has complete, readable JavaScript — not bytecode stubs. Patches can match and replace patterns in actual function bodies.

**Why our approach failed**: Our extracted source starts with `// @bun @bytecode @bun-cjs` and contains compiled stubs. Even after stripping the prefix and clearing bytecode, the source is not interpretable by Bun as plain JS because the function bodies are placeholders, not real code.

## The Actual Fix Needed

Implement the tweakcc PR #655 approach:

1. **Detect bytecode compilation**: Check if extracted content starts with `// @bun @bytecode`
2. **Fetch clean source**: Run `npm pack @anthropic-ai/claude-code@{version}` to get readable source
3. **Patch clean source**: Apply all patches to the npm-fetched readable `cli.js`
4. **Repack with bytecode cleared**: Use the patched readable source + `Buffer.alloc(0)` for bytecode + `encoding=0`

This matches exactly what tweakcc implemented and verified working.
