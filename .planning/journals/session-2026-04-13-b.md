# Session 2026-04-13-b — Phase 2c Research & Scoping

## What Happened

Post-compaction session. Previous session completed Phase 2b-gaps-3 (REPL replace mode hardening). This session researched and scoped Phase 2c (Clean-Room Tungsten).

## Key Decisions

1. **Tungsten is a persistent execution context, not just a tmux tool.** First-principles analysis: CC's fundamental limitation is turn-based interaction with a continuous OS. Tungsten bridges that gap — persistent sessions, surviving processes, stateful shell. The tool is the control surface; the real value is environment persistence.

2. **Two-part delivery: tool + binary patch.** The tool alone gives explicit session management. The FS9() binary patch unlocks bashProvider tmux integration, making ALL Bash calls (including REPL's bash() delegation) inherit the tmux context. Both are needed for the complete experience.

3. **UI injection is feasible.** TungstenLiveMonitor and all UI code is fully DCE'd, but the render tree has an identifiable injection point (`!1,null` at offset 11998161). A binary patch can replace it with a createElement call that loads our panel component. React hooks (useAppState) survive and the tool can write to AppState via context.setAppState().

4. **6 deliverables, nothing deferred:** D1 (tungsten.js), D2 (FS9 binary patch), D3 (render tree injection), D4 (tungsten-panel.js), D5 (statusline + hooks), D6 (REPL prompt update).

## Key Findings

- **FS9() = getClaudeTmuxEnv()** — stubbed to `return null` in binary. The bashProvider plumbing is intact: it calls FS9(), and if non-null, sets TMUX env on all Bash commands. Patching FS9 to return our socket info activates the entire chain.
- **`!1,null` at offset 11998161** — the DCE'd TungstenLiveMonitor in the render tree. Unique signature: `cn7(O_)))),!1,null,b_.createElement(m,{flexGrow:1})`
- **useAppState (Y_) survives** — Zustand-like hook via useSyncExternalStore. Component injection can receive it as a prop.
- **All tungsten AppState fields DCE'd** — tungstenActiveSession, tungstenLastCommand, etc. all zero occurrences. But AppState is an untyped JS object at runtime — any key works.
- **REPL agent() works** — smoke tested via `claude -p`. Delegates to native Agent tool through stash, returns result.
- **Zero "Tungsten" or "Tmux" tool name in binary** — complete DCE of the tool wrapper, consistent with it being a thin layer over tmuxSocket.ts infrastructure.
- **tmuxSocket.ts functions DCE'd** — hasTmuxToolBeenUsed, ensureSocketInitialized, markTmuxToolUsed, getClaudeTmuxEnv — all zero. Only FS9 (the stub) and "tmux" strings survive.
- **Agent swarm tmux code survives** — new-session, break-pane etc. are for swarm infrastructure, not Tungsten.
